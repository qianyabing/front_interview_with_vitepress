## 🎯 一、明确面试官的考察意图：不只是背八股文

面试官考察设计模式，真正关心的是以下三点：

1. **你是否理解问题的本质**：能识别业务场景中的痛点，并判断该用什么模式解决
2. **你是否能选择合适的模式**：不是硬套模式，而是根据场景权衡利弊

**核心原则**：设计模式是工具，而不是目的。在面试中展示你如何**根据实际问题选择合适的模式**，这才是面试官真正想看到的。

## 🗂️ 二、高频设计模式：按优先级分层总结

### 📌 第一优先级：创建型模式（大厂最爱考）

大厂面试中考察的重点就是**创建型设计模式**，比如工厂模式、抽象工厂模式和单例模式，是校招和社招中比较常见的问题。创建型模式主要解决「如何创建对象」的问题，简化创建过程，降低耦合度。

| 模式 | 一句话理解 | 前端应用场景 | 手写实现要点 |
| --- | --- | --- | --- |
| **单例模式** | 全局只有一个实例 | Redux/Vuex的Store、全局弹窗管理、日志记录器 | 用闭包或静态属性缓存实例，重复创建时返回已有实例 |
| **工厂模式** | 用一个函数专门负责创建对象 | 根据类型创建不同UI组件、Axios实例化、不同权限的菜单 | 封装new逻辑，统一创建入口 |
| **建造者模式** | 分步骤构建复杂对象 | 复杂配置对象的构建（如请求参数组装） | 链式调用 + 逐步构建 |

**💡 单例模式懒汉/饿汉区别**（Java体系面试常延伸）：

- **饿汉模式**：类初始化时就创建实例，线程安全，但可能造成资源浪费
- **懒汉模式**：首次使用时才创建实例，需注意线程安全问题（双检锁）

---

### 📌 第二优先级：行为型模式（框架原理高频）

行为型模式主要解决「对象之间怎么协作通信」的问题。大厂特别喜欢考察这些模式与Vue/React框架的结合理解。

| 模式 | 一句话理解 | 前端应用场景 | 面试高频追问 |
| --- | --- | --- | --- |
| **观察者模式 vs 发布-订阅模式** | 观察者直接通知订阅者 vs 通过事件中心解耦 | Vue响应式系统（观察者）、EventBus（发布-订阅） | **区别是什么？** 观察者中观察者和主体知道彼此；发布-订阅中通过消息代理通信，耦合度更低 |
| **策略模式** | 定义一系列算法，封装起来可以互换 | 表单验证规则、支付方式处理、排序算法切换 | 避免大量if-else，符合开闭原则 |
| **模板方法模式** | 定义算法骨架，子类重写特定步骤 | 框架生命周期钩子、通用流程封装 | 好莱坞原则：「别调用我们，我们会调用你」 |

---

### 📌 第三优先级：结构型模式（加分项）

结构型模式主要解决「对象之间怎么组织结构」的问题。

| 模式 | 一句话理解 | 前端应用场景 |
| --- | --- | --- |
| **代理模式** | 为对象找个中间人控制访问 | Vue3的Proxy响应式系统、图片懒加载、缓存代理 |
| **装饰器模式** | 动态添加功能，不改变原结构 | React高阶组件(HOC)、日志/缓存装饰、ES7装饰器语法 |
| **适配器模式** | 兼容接口不匹配的双方 | 适配不同API响应格式、集成第三方库 |

## 📚 三、知识体系构建方法

### 🔹 Step 1：建立三大分类框架

将23种设计模式按**创建型、结构型、行为型**三大类系统学习：

| 类型 | 数量 | 代表模式 |
| --- | --- | --- |
| **创建型** | 5种 | 工厂方法、抽象工厂、单例、建造者、原型 |
| **结构型** | 7种 | 适配器、装饰器、代理、外观、桥接、组合、享元 |
| **行为型** | 11种 | 策略、模板方法、观察者、迭代子、责任链、命令、备忘录、状态、访问者、中介者、解释器 |

### 🔹 Step 2：理解六大设计原则（SOLID）

| 原则 | 核心要点 | 面试话术 |
| --- | --- | --- |
| **单一职责** | 一个类/函数只做一件事 | 「我设计模块时会确保每个类只有一个变更理由，降低耦合」 |
| **开放封闭** | 对扩展开放，对修改封闭 | 「新增功能时优先扩展而非修改已有代码，避免引入Bug」 |
| **里氏替换** | 子类不改变父类功能 | 「继承时确保子类能完全替代父类」 |
| **依赖倒置** | 面向接口编程，不依赖具体实现 | 「我会依赖抽象而非具体类，提高模块的可替换性」 |

### 🔹 Step 3：总结每个模式的「三要素」

对每个高频模式，按照「**定义 → 应用场景 → 代码实现**」三要素来记忆和总结：

```text
示例：单例模式
├── 定义：确保一个类只有一个实例，并提供全局访问点
├── 应用场景：全局状态管理（Redux Store）、弹窗管理器、日志记录器
└── 代码实现：闭包或静态属性缓存实例
```

## 🧱 第一部分：创建型模式（共3讲）

### 第1讲：单例模式 —— “唯一的我”

#### 📌 一句话理解

> 一个类**只能有一个实例**，并提供一个全局访问点。

#### 🎯 真实前端场景

- **全局弹窗**（Toast、Modal）：无论调用多少次，页面上只有一个弹窗实例，避免重复创建。
- **Vuex / Redux 的 Store**：整个应用只有一个数据仓库。
- **日志记录器**：所有地方打日志都往同一个对象里写。

#### 💻 手写一个最简单例（JavaScript）

```javascript
const Singleton = (function() {
  let instance = null;
  function createInstance() {
    return { name: '我是唯一的实例' };
  }
  return {
    getInstance: function() {
      if (!instance) instance = createInstance();
      return instance;
    }
  }
})();

const a = Singleton.getInstance();
const b = Singleton.getInstance();
console.log(a === b); // true
```

#### ❓ 面试高频追问

**Q：单例模式有什么缺点？**
A：全局状态可能导致模块间**隐式耦合**，测试时难以隔离；且在多线程环境（Node.js worker）下需要额外注意。

**Q：前端单例一定用闭包吗？ES6 怎么简洁实现？**
A：也可以用 ES6 模块的特性——**模块本身就是单例**：

```javascript
// logger.js
export default { log: msg => console.log(msg) };
// 任何地方 import 的都是同一个实例
```

**Q：懒汉 vs 饿汉？**
A：前端几乎都是“懒汉”（用到才创建），因为页面加载时不确定是否需要；饿汉（一开始就创建）适合资源固定且必定会用到的场景。

---

### 第2讲：工厂模式 —— “你要什么，我给你造”

#### 📌 一句话理解

> 用一个**工厂函数**来创建对象，不用每次自己 `new`，尤其适合**根据不同类型创建不同对象**。

#### 🎯 真实前端场景

- **不同权限的按钮**：管理员看到“删除”按钮，普通用户看到“查看”按钮。
- **不同消息类型组件**：成功、警告、错误消息用不同的样式和图标。
- **Axios 实例工厂**：根据 baseURL 或 token 创建不同的请求实例。

#### 💻 简单工厂（最常用）

```javascript
class ButtonAdmin { render() { return '管理员按钮' } }
class ButtonUser { render() { return '普通按钮' } }

class ButtonFactory {
  createButton(role) {
    switch(role) {
      case 'admin': return new ButtonAdmin();
      case 'user': return new ButtonUser();
      default: throw new Error('未知角色');
    }
  }
}

const factory = new ButtonFactory();
const btn = factory.createButton('admin');
```

#### ❓ 面试高频追问

**Q：工厂模式和直接 `new` 有什么区别？**
A：工厂将**创建逻辑集中**，调用方不需要知道具体类名，也方便后续扩展（比如增加 `superAdmin` 按钮只需改工厂，调用方无感知）。

**Q：简单工厂、工厂方法、抽象工厂的区别？**
A：

- **简单工厂**：一个工厂生产所有产品（if-else 或 switch）。
- **工厂方法**：每种产品对应一个工厂子类，符合开闭原则，但类会增多。
- **抽象工厂**：生产一系列相关产品（例如一套 UI 主题包含按钮、输入框、对话框）。

前端最常用的是**简单工厂**和**工厂方法**，抽象工厂在主题切换库（如 antd 的样式主题）中有体现。

## 🧱 第一部分：创建型模式（第3讲）

### 第3讲：建造者模式 —— “分步骤造一个复杂对象”

#### 📌 一句话理解

> 把**复杂对象的构建过程拆成多个小步骤**，可以按不同顺序或配置组装，最终得到不同表现的对象。

#### 🎯 真实前端场景

- **复杂配置对象的生成**：比如构造一个报表请求参数，有分页、排序、筛选、统计等多个可选部分。
- **表单校验器链**：一步步添加校验规则。
- **URL 参数构建器**：逐步添加 query 参数。

#### 💻 手写一个前端建造者示例

```javascript
class RequestBuilder {
  constructor() {
    this.url = '';
    this.method = 'GET';
    this.headers = {};
    this.body = null;
  }
  setUrl(url) { this.url = url; return this; }      // 返回 this 支持链式调用
  setMethod(method) { this.method = method; return this; }
  setHeader(key, value) { this.headers[key] = value; return this; }
  setBody(body) { this.body = body; return this; }
  build() {
    // 可以在这里做参数校验
    if (!this.url) throw new Error('URL is required');
    return {
      url: this.url,
      method: this.method,
      headers: this.headers,
      body: this.body
    };
  }
}

// 使用
const request = new RequestBuilder()
  .setUrl('/api/user')
  .setMethod('POST')
  .setHeader('Content-Type', 'application/json')
  .setBody({ name: 'John' })
  .build();
```

#### ❓ 面试高频追问

**Q：建造者模式 vs 工厂模式的区别？**
A：

- **工厂模式**：一把梭，直接产出完整对象，适合创建逻辑简单或固定结构的对象。
- **建造者模式**：分步构建，适合对象结构复杂、参数可选、不同组合差异大的场景。

> 面试金句：*「工厂模式像快餐店点套餐，直接拿；建造者模式像自助餐，自己一步步选。」*

**Q：前端真的需要建造者模式吗？**
A：传统面向对象语言（Java）中常见，但前端可以用对象字面量 + 默认值替代。不过当**构建步骤很多且有顺序依赖**时，建造者模式让代码更清晰。Vue 源码中的 `config` 合并策略也可看作一种变体。

## 🧰 第二部分：结构型模式（第1讲）

### 第1讲：代理模式 —— “找个中间人帮你做事”

#### 📌 一句话理解

> 提供一个**替身或占位符**来控制对原对象的访问，可以在访问前后做额外操作（比如缓存、校验、日志、懒加载）。

#### 🎯 真实前端场景（大厂必考）

- **Vue3 响应式系统**：用 `Proxy` 代理对象，拦截 get/set 实现依赖收集和视图更新。
- **图片懒加载**：先用一个占位图代理真实图片，滚动到可视区再加载。
- **缓存代理**：缓存计算结果，避免重复请求或计算。
- **权限校验代理**：在发送请求前检查是否有 token。

#### 💻 手写三种常见代理

**1. 缓存代理（面试手写高频）**

```javascript
// 原函数：计算斐波那契（耗时）
function fib(n) {
  if (n < 2) return n;
  return fib(n - 1) + fib(n - 2);
}

// 代理：加缓存
const fibProxy = (function() {
  const cache = {};
  return function(n) {
    if (cache[n] !== undefined) return cache[n];
    cache[n] = fib(n);
    return cache[n];
  }
})();
```

**2. Vue3 响应式代理（简化版）**

```javascript
function reactive(obj) {
  return new Proxy(obj, {
    get(target, key) {
      console.log(`读取 ${key}`);
      return target[key];
    },
    set(target, key, value) {
      console.log(`设置 ${key} = ${value}`);
      target[key] = value;
      // 触发更新（实际调用 effect）
      return true;
    }
  });
}
```

**3. 图片懒加载代理**

```html
<img data-src="real.jpg" src="placeholder.jpg" />
```

```javascript
const imgProxy = {
  loadRealImg(imgElement) {
    const realSrc = imgElement.dataset.src;
    const img = new Image();
    img.onload = () => { imgElement.src = realSrc; };
    img.src = realSrc;
  }
};
```

#### ❓ 面试高频追问

**Q：代理模式 vs 装饰器模式的区别？**
A：这是大厂常问的辨析题。

- **代理模式**：控制访问，强调**对目标对象的访问管理**（缓存、校验、懒加载），接口通常不变。
- **装饰器模式**：动态添加功能，强调**增强行为**（加日志、加格式转换），会一层层包装。

> 举例：代理像私人助理帮你过滤电话；装饰器像手机壳，套一层加个支架。

**Q：`Proxy` 和 `Object.defineProperty` 实现响应式的区别？**
A：

- `defineProperty`：只能监听已有属性，需要递归遍历，无法监听数组索引和 length 变化。
- `Proxy`：监听整个对象，包括新增属性和数组变化，性能更好，代码更简洁。

## ⚙️ 第三部分：行为型模式（第1讲）

### 第1讲：观察者模式 vs 发布-订阅模式 —— “你喊一声，我响应”

> 这是面试中的**绝对高频考点**，尤其是结合 Vue 响应式原理来问。

#### 📌 一句话理解

- **观察者模式**：目标对象（Subject）直接通知所有观察者（Observer），大家互相认识。
- **发布-订阅模式**：发布者（Publisher）和订阅者（Subscriber）互不认识，通过**事件中心（Event Bus）** 通信。

#### 🎯 真实前端场景

| 模式 | 场景举例 |
| --- | --- |
| **观察者模式** | Vue 响应式系统（Dep 通知 Watcher）、DOM 事件监听（addEventListener） |
| **发布-订阅模式** | EventBus、Vue 的 `$on`/`$emit`、Node.js 的 EventEmitter、消息队列 |

#### 💻 手写实现（面试必会）

**1. 观察者模式**

```javascript
// 被观察者（目标）
class Subject {
  constructor() {
    this.observers = [];
  }
  addObserver(observer) {
    this.observers.push(observer);
  }
  notify(data) {
    this.observers.forEach(observer => observer.update(data));
  }
}

// 观察者
class Observer {
  constructor(name) {
    this.name = name;
  }
  update(data) {
    console.log(`${this.name} 收到通知:`, data);
  }
}

// 使用
const sub = new Subject();
const obs1 = new Observer('观察者A');
const obs2 = new Observer('观察者B');
sub.addObserver(obs1);
sub.addObserver(obs2);
sub.notify('下雨了');
```

**2. 发布-订阅模式（事件中心）**

```javascript
class EventBus {
  constructor() {
    this.events = {};
  }
  on(eventName, callback) {
    if (!this.events[eventName]) this.events[eventName] = [];
    this.events[eventName].push(callback);
  }
  emit(eventName, data) {
    const callbacks = this.events[eventName];
    if (callbacks) callbacks.forEach(cb => cb(data));
  }
  off(eventName, callback) {
    if (!callback) delete this.events[eventName];
    else {
      const idx = this.events[eventName]?.indexOf(callback);
      if (idx !== -1) this.events[eventName].splice(idx, 1);
    }
  }
}

// 使用
const bus = new EventBus();
bus.on('login', (user) => console.log(`欢迎 ${user}`));
bus.emit('login', '张三');
```

#### ❓ 面试高频追问（重要！）

**Q1：两者的核心区别是什么？**

| 对比点 | 观察者模式 | 发布-订阅模式 |
| --- | --- | --- |
| **耦合度** | 目标和观察者互相知道，**松耦合但仍有依赖** | 完全解耦，通过事件中心中介 |
| **角色** | Subject + Observer | Publisher + Event Channel + Subscriber |
| **灵活性** | 同步通知，通常按顺序执行 | 支持异步、消息持久化、更灵活 |
| **典型实现** | Vue 响应式、RxJS 的 Observable | EventBus、RabbitMQ、浏览器事件 |

> 面试金句：*「观察者模式是‘面对面’通知，发布-订阅是‘对讲机’广播。」*

**Q2：Vue 响应式用的是观察者还是发布-订阅？**
A：两者都有。Vue 的**响应式核心是观察者模式**：`Dep`（目标）收集 `Watcher`（观察者），数据变化时 `Dep` 通知 `Watcher` 更新。但组件通信的 `$on`/`$emit` 是发布-订阅模式。

**Q3：为什么 Vue 不直接用发布-订阅做响应式？**
A：因为响应式需要**精确的依赖收集**（哪个组件用到了这个数据就通知谁），观察者模式更直接高效；发布-订阅适合跨组件、跨模块的松散通信。

### 第2讲：策略模式 —— “消灭 if-else 的神器”

#### 📌 一句话理解

> 定义**一族算法**，把它们封装起来，让它们可以互相替换。调用方根据需要选择策略，而不是写死逻辑。

#### 🎯 真实前端场景

- **表单验证**：不同字段用不同验证规则（非空、手机号、邮箱…）
- **年终奖计算**：不同绩效等级用不同系数
- **动画缓动函数**：linear、ease-in、ease-out
- **支付方式**：微信、支付宝、银行卡

#### 💻 手写示例

**传统 if-else（坏味道）**

```javascript
function getSalary(level, baseSalary) {
  if (level === 'S') return baseSalary * 3;
  else if (level === 'A') return baseSalary * 2;
  else if (level === 'B') return baseSalary * 1.5;
  else return baseSalary;
}
```

**策略模式重构**

```javascript
// 策略对象
const salaryStrategies = {
  S: (base) => base * 3,
  A: (base) => base * 2,
  B: (base) => base * 1.5,
  C: (base) => base
};

function getSalary(level, baseSalary) {
  const strategy = salaryStrategies[level];
  return strategy ? strategy(baseSalary) : baseSalary;
}

// 甚至可以动态添加策略
salaryStrategies['SSS'] = (base) => base * 5;
```

**表单验证例子**

```javascript
const validators = {
  isNotEmpty: (val) => val.trim() !== '',
  isEmail: (val) => /^[^\s@]+@[^\s@]+\.[^\s@]+$/.test(val),
  minLength: (len) => (val) => val.length >= len
};

function validate(value, rules) {
  for (let rule of rules) {
    let result;
    if (typeof rule === 'function') result = rule(value);
    else if (rule.name) result = validators[rule.name](value);
    if (!result) return false;
  }
  return true;
}

// 使用
validate('abc', [validators.isNotEmpty, validators.minLength(3)]); // true
```

### 第3讲：模板方法模式 —— “我定流程，你来填空”

#### 📌 一句话理解

> 在父类中定义一个算法的**骨架**（模板），把一些具体步骤延迟到子类中实现。子类可以不改变算法结构，只重写某些步骤。

#### 🎯 真实前端场景

- **框架生命周期钩子**：Vue/React 的 `mounted`、`updated` 等，框架定好流程，你在钩子里填逻辑。
- **通用请求流程**：前置拦截 → 发请求 → 后置处理 → 错误处理，子类可重写拦截或错误处理。
- **组件渲染流程**：`beforeRender → render → afterRender`，不同组件实现自己的渲染细节。

#### 💻 手写示例

**1. 通用请求模板**

```javascript
// 抽象模板（父类）
class RequestTemplate {
  // 模板方法：定义请求流程骨架
  async request(url, options) {
    const config = this.beforeRequest(options);
    try {
      const data = await this.doRequest(url, config);
      return this.afterRequest(data);
    } catch (error) {
      return this.handleError(error);
    }
  }
  // 钩子方法（子类可重写）
  beforeRequest(options) { return options; }
  afterRequest(data) { return data; }
  handleError(error) { throw error; }
  // 必须由子类实现的方法
  doRequest(url, options) {
    throw new Error('子类必须实现 doRequest');
  }
}

// 具体子类：JSON 请求
class JSONRequest extends RequestTemplate {
  beforeRequest(options) {
    return {
      ...options,
      headers: { 'Content-Type': 'application/json', ...options.headers }
    };
  }
  async doRequest(url, options) {
    const res = await fetch(url, options);
    return res.json();
  }
  handleError(error) {
    console.error('JSON请求失败:', error);
    return null;
  }
}
```

**2. Vue 生命周期类比**

```javascript
// 框架内部伪代码
class VueComponent {
  _mount() {               // 模板方法
    this.beforeCreate();
    this.createData();
    this.beforeMount();
    this.render();
    this.mounted();
  }
  beforeCreate() {}
  createData() { /* 实现 */ }
  beforeMount() {}
  render() { throw new Error('must implement') }
  mounted() {}
}
// 开发者写组件时只需重写钩子
```

#### ❓ 面试高频追问

**Q：模板方法模式的核心思想是什么？**
A：**好莱坞原则**——「Don't call us, we'll call you」。框架控制流程，调用你提供的钩子。这能实现**复用不变部分，定制变化部分**。

**Q：和策略模式的区别？**
A：

- **模板方法**：在继承层次中，父类定骨架，子类填空（横向扩展）
- **策略模式**：通过组合，把算法抽离成独立策略对象（纵向扩展）

> 举例：
>
> - 模板方法：做菜流程固定（洗→切→炒→装盘），但你可以决定放什么调料。
> - 策略模式：今天选「川菜策略」还是「粤菜策略」，整体替换一套做法。

**Q：前端框架中哪里体现了模板方法？**
A：Vue 的 `options` 合并策略、生命周期钩子；React 类组件的生命周期方法；Web Component 的 `connectedCallback` 等。本质上框架都遵循「骨架 + 钩子」的模式。

### 第4讲：责任链模式 —— “层层上报，直到有人处理”

#### 📌 一句话理解

> 将请求沿着一条链传递，链上的每个处理器**决定处理还是转发**，直到某个处理器处理掉它。

#### 🎯 真实前端场景

- **中间件机制**：Express / Koa 的 `app.use()`，请求一层层经过中间件。
- **表单校验**：非空 → 格式校验 → 长度校验 → 自定义规则，任何一个失败就拦截。
- **权限审批流**：员工请假 → 组长 → 经理 → HR。
- **Axios 拦截器**：请求拦截器链和响应拦截器链。

#### 💻 手写示例

**1. 简单责任链（校验器）**

```javascript
// 处理器基类
class ValidatorHandler {
  setNext(handler) {
    this.next = handler;
    return handler;  // 方便链式调用
  }
  handle(request) {
    if (this.next) return this.next.handle(request);
    return { success: true };
  }
}

// 具体处理器
class NotEmptyValidator extends ValidatorHandler {
  handle(request) {
    if (!request.value || request.value.trim() === '') {
      return { success: false, msg: '不能为空' };
    }
    return super.handle(request);
  }
}

class MinLengthValidator extends ValidatorHandler {
  constructor(min) { super(); this.min = min; }
  handle(request) {
    if (request.value.length < this.min) {
      return { success: false, msg: `长度不能小于${this.min}` };
    }
    return super.handle(request);
  }
}

// 使用
const notEmpty = new NotEmptyValidator();
const minLength = new MinLengthValidator(3);
notEmpty.setNext(minLength);
console.log(notEmpty.handle({ value: 'ab' }));   // { success: false, msg: '长度不能小于3' }
console.log(notEmpty.handle({ value: 'abc' }));  // { success: true }
```

**2. 模拟 Koa 中间件（洋葱模型简化版）**

```javascript
class MiddlewareChain {
  constructor() { this.middlewares = []; }
  use(fn) { this.middlewares.push(fn); return this; }
  async run(ctx) {
    let index = -1;
    const dispatch = async (i) => {
      if (i <= index) throw new Error('next() called multiple times');
      index = i;
      const fn = this.middlewares[i];
      if (!fn) return;
      await fn(ctx, () => dispatch(i + 1));
    };
    await dispatch(0);
    return ctx;
  }
}

// 使用
const chain = new MiddlewareChain();
chain.use(async (ctx, next) => {
  console.log('日志开始');
  await next();
  console.log('日志结束');
});
chain.use(async (ctx, next) => {
  ctx.body = 'Hello';
  await next();
});
chain.run({}).then(console.log);
// 输出：日志开始 → (无更多中间件) → 日志结束 → { body: 'Hello' }
```

#### ❓ 面试高频追问

**Q：责任链模式有什么优缺点？**
A：

- ✅ **优点**：解耦发送者和接收者；可以动态组合链；增加新处理器方便。
- ❌ **缺点**：请求可能无人处理；调试时链路过长不易追踪；性能有损耗。

**Q：责任链模式和装饰器模式的区别？**
A：

- **责任链**：每个节点**选择处理或传递**，通常只被一个节点处理。
- **装饰器**：每个装饰器**都会执行**，层层增强。

> 例子：
>
> - 责任链：请假申请，组长批了就不再到经理。
> - 装饰器：给蛋糕加奶油、加水果、加巧克力，所有装饰都生效。

**Q：前端哪些库用了责任链？**
A：Express/Koa 中间件、Axios 拦截器、各种表单校验库、Promise 的 `then/catch` 链（也可以看作责任链的变体）。

### 第5讲：迭代器模式 —— “只管遍历，不管内部”

#### 📌 一句话理解

> 提供一种**统一的方法**来顺序访问一个集合对象的内部元素，而不需要暴露该集合的内部结构。

#### 🎯 真实前端场景

- **JavaScript 原生迭代器**：`for...of`、`Array.prototype.map/forEach`、`Set`/`Map` 的遍历
- **手写迭代器**：自定义数据结构需要被遍历时（如树、链表、图）
- **分页遍历**：后端分页数据，封装成可迭代的接口

#### 💻 手写示例

**1. 原生迭代器协议**

```javascript
// 任何实现了 Symbol.iterator 方法的对象都可以被 for...of 遍历
const myCollection = {
  items: [10, 20, 30],
  [Symbol.iterator]() {
    let index = 0;
    const items = this.items;
    return {
      next() {
        if (index < items.length) {
          return { value: items[index++], done: false };
        } else {
          return { done: true };
        }
      }
    };
  }
};
for (let val of myCollection) {
  console.log(val); // 10 20 30
}
```

**2. 树结构的迭代器（深度优先）**

```javascript
class TreeNode {
  constructor(value, children = []) {
    this.value = value;
    this.children = children;
  }
  // 实现迭代器：深度优先遍历
  [Symbol.iterator]() {
    const stack = [this];
    return {
      next() {
        while (stack.length) {
          const node = stack.pop();
          // 子节点逆序推入，保证正序遍历顺序
          for (let i = node.children.length - 1; i >= 0; i--) {
            stack.push(node.children[i]);
          }
          return { value: node.value, done: false };
        }
        return { done: true };
      }
    };
  }
}
const tree = new TreeNode(1, [
  new TreeNode(2, [new TreeNode(4), new TreeNode(5)]),
  new TreeNode(3)
]);
for (let val of tree) console.log(val); // 1 2 4 5 3
```

**3. 分页迭代器（包装后端分页API）**

```javascript
class PageIterator {
  constructor(fetchPage) {  // fetchPage 接收页码，返回 Promise<{data, hasMore}>
    this.fetchPage = fetchPage;
    this.currentPage = 1;
    this.hasMore = true;
  }
  async next() {
    if (!this.hasMore) return { done: true };
    const { data, hasMore } = await this.fetchPage(this.currentPage);
    this.hasMore = hasMore;
    this.currentPage++;
    return { value: data, done: false };
  }
  // 使其可异步迭代
  [Symbol.asyncIterator]() {
    return this;
  }
}
// 使用
const iter = new PageIterator((page) => fetch(`/api/list?page=${page}`).then(r => r.json()));
for await (let pageData of iter) {
  console.log(pageData);
}
```

**Q：迭代器模式解决了什么问题？**
A：解耦**遍历算法**和**数据结构**。不管数据是数组、树还是数据库游标，调用方都用统一的 `next()` 或 `for...of` 来遍历。

**Q：迭代器模式和 Generator 函数的关系？**
A：Generator 函数返回的对象天然实现了迭代器协议，是写迭代器的语法糖。

```javascript
function* range(start, end) {
  for (let i = start; i <= end; i++) yield i;
}
for (let num of range(1, 5)) console.log(num);
```

### 第6讲：状态模式 —— “状态变了，行为就变”

#### 📌 一句话理解

> 允许一个对象在其**内部状态改变**时改变它的行为，看起来就像修改了它的类。

#### 🎯 真实前端场景

- **有限状态机**：Promise（pending → fulfilled/rejected）、红绿灯、电梯状态
- **组件状态管理**：加载中 → 成功 → 失败 → 重试
- **游戏角色状态**：站立、跑动、跳跃、倒地，每个状态有不同的行为逻辑
- **XState 库**：专门用于状态机

#### 💻 手写示例

**1. 传统 if-else 写法（坏味道）**

```javascript
class TrafficLight {
  constructor() { this.state = 'red'; }
  change() {
    if (this.state === 'red') {
      console.log('红灯 → 绿灯');
      this.state = 'green';
    } else if (this.state === 'green') {
      console.log('绿灯 → 黄灯');
      this.state = 'yellow';
    } else if (this.state === 'yellow') {
      console.log('黄灯 → 红灯');
      this.state = 'red';
    }
  }
}
```

**2. 状态模式重构**

```javascript
// 状态接口（抽象）
class LightState {
  constructor(light) { this.light = light; }
  change() { throw new Error('子类实现'); }
}

// 具体状态类
class RedState extends LightState {
  change() {
    console.log('红灯 → 绿灯');
    this.light.setState(this.light.greenState);
  }
}
class GreenState extends LightState {
  change() {
    console.log('绿灯 → 黄灯');
    this.light.setState(this.light.yellowState);
  }
}
class YellowState extends LightState {
  change() {
    console.log('黄灯 → 红灯');
    this.light.setState(this.light.redState);
  }
}

// 上下文（维护状态）
class TrafficLight {
  constructor() {
    this.redState = new RedState(this);
    this.greenState = new GreenState(this);
    this.yellowState = new YellowState(this);
    this.state = this.redState;
  }
  setState(state) { this.state = state; }
  change() { this.state.change(); }
}

// 使用
const light = new TrafficLight();
light.change(); // 红灯→绿灯
light.change(); // 绿灯→黄灯
light.change(); // 黄灯→红灯
```

**3. 更简洁的实现（函数式 + 策略表）**

```javascript
const stateMachine = {
  state: 'red',
  transitions: {
    red: { change: () => { console.log('→绿'); return 'green'; } },
    green: { change: () => { console.log('→黄'); return 'yellow'; } },
    yellow: { change: () => { console.log('→红'); return 'red'; } }
  },
  change() {
    const transition = this.transitions[this.state];
    if (transition && transition.change) {
      this.state = transition.change();
    }
    return this;
  }
};
stateMachine.change().change().change();
```

#### ❓ 面试高频追问

**Q：状态模式和策略模式有什么区别？**
A：两者结构相似，但**意图不同**：

| 对比点 | 状态模式 | 策略模式 |
| --- | --- | --- |
| **核心** | 状态变化导致行为变化 | 算法可互换，但对象状态不变 |
| **状态切换** | 状态通常由上下文或状态自己决定 | 策略由调用方主动选择 |
| **关注点** | 对象在不同状态下不同行为 | 同一任务的不同算法实现 |

> 例子：
>
> - 状态模式：订单状态（待支付→已支付→已发货→已完成），每个状态下取消订单的行为不同。
> - 策略模式：支付方式（微信、支付宝、银行卡），算法可替换但订单状态不变。

**Q：前端状态管理（Vuex/Redux）和状态模式的关系？**
A：状态模式是设计层面**对象内部状态机**；Vuex/Redux 是全局状态管理，但其中的 reducer 本质上也是一个状态机（根据 action type 和当前 state 返回新 state）。

**Q：什么场景该用状态模式？**
A：当一个对象的行为**严重依赖于它的状态**，并且状态数量较多、状态转换频繁时。比如表单的多步骤流程、播放器的播放/暂停/停止状态。

## 🔥 四大高频设计模式「串烧记忆卡」

> 适合面试前 10 分钟快速回顾，每个模式用 **一句话核心 + 记忆锚点 + 手写骨架 + 注意点** 串起来。

---

## 1️⃣ 单例模式（Singleton）

- **一句话核心**：保证一个类**只有一个实例**，并提供全局访问点。
- **记忆锚点**：全局弹窗、Vuex Store、登录态管理。
- **手写骨架**（闭包懒汉）：
  ```javascript
  const Singleton = (function() {
  let instance = null;
  return { getInstance: () => instance || (instance = {}) };
  })();
  ```
- **ES6 模块版**：直接 export 一个对象，模块系统天然单例。
- **注意点**：
  - 不要滥用 —— 全局状态会增加隐式耦合，测试困难。
  - 前端几乎没有“饿汉模式”（页面加载时就创建），因为不一定用到。
  - 多例场景（如需要多个独立弹窗）不要强行用单例。

---

## 2️⃣ 工厂模式（Factory）

- **一句话核心**：用一个**工厂函数**来创建对象，调用方不用自己 `new`，根据类型产出不同实例。
- **记忆锚点**：根据权限生成不同按钮、Axios 实例工厂、消息提示（success/warning/error）。
- **手写骨架**（简单工厂）：
  ```javascript
  class ButtonFactory {
  create(role) {
    switch(role) {
      case 'admin': return new AdminButton();
      case 'user': return new UserButton();
      default: return new GuestButton();
    }
  }
  }
  ```
- **注意点**：
  - 简单工厂适合产品类型少的场景；类型多时考虑工厂方法（每种产品一个工厂子类）。
  - 工厂模式的核心价值是**隔离创建逻辑**，增加新产品时调用方无感知。
  - 不要为了用工厂而用 —— 如果 `new` 一个对象很简单且不会变化，直接 `new` 即可。

---

## 3️⃣ 观察者模式（Observer）

- **一句话核心**：**目标对象（Subject）** 维护一组观察者（Observer），状态变化时直接通知所有观察者。
- **记忆锚点**：Vue 响应式系统（Dep → Watcher）、DOM 事件监听。
- **手写骨架**：
  ```javascript
  class Subject {
  constructor() { this.obs = []; }
  add(o) { this.obs.push(o); }
  notify(data) { this.obs.forEach(o => o.update(data)); }
  }
  class Observer {
  update(data) { console.log(data); }
  }
  ```
- **注意点**：
  - **与发布-订阅模式的区分**：观察者模式中目标和观察者互相知道（耦合），发布-订阅通过事件中心完全解耦。
  - 面试高频追问：Vue 响应式是观察者模式，`$on/$emit` 是发布-订阅。
  - 注意内存泄漏 —— 观察者没有被正确移除会导致无法释放。

---

## 4️⃣ 策略模式（Strategy）

- **一句话核心**：定义一组**可互换的算法**，把它们封装成独立策略，调用方根据需要选择策略。
- **记忆锚点**：表单验证（不同字段不同规则）、年终奖计算、动画缓动函数。
- **手写骨架**：
  ```javascript
  const strategies = {
  S: salary => salary * 3,
  A: salary => salary * 2,
  B: salary => salary * 1.5
  };
  function getBonus(level, salary) {
  return strategies[level]?.(salary) ?? salary;
  }
  ```
- **注意点**：
  - 核心价值是**消灭 if-else / switch**，符合开闭原则。
  - 策略模式通常与**简单工厂**结合使用（工厂负责创建哪个策略，策略负责执行算法）。
  - 不要过度设计 —— 如果只有 1-2 个简单条件，用 if-else 更直观。

| 模式 | 类型 | 解决什么问题 | 一句话类比 |
| --- | --- | --- | --- |
| 单例 | 创建型 | 全局唯一实例 | 公司只有一个 CEO |
| 工厂 | 创建型 | 统一创建对象，隐藏 `new` | 奶茶店根据你点的口味做饮料 |
| 观察者 | 行为型 | 状态变化时自动通知依赖方 | 你关注了博主，他发视频你会收到通知 |
| 策略 | 行为型 | 算法可互换，避免条件分支 | 地图 App 让你选：驾车/公交/步行策略 |

## ⚠️ 面试常见踩坑点（避雷区）

1. **单例**：不要说“用 `export default {}` 就是单例” —— 是的，但那是模块特性，不是模式本身；最好同时会写闭包版。
2. **工厂**：不要把工厂写成巨大的 if-else 怪物 —— 那是“简单工厂”的缺点，可以提“工厂方法”改进。
3. **观察者**：不要和发布-订阅混为一谈 —— 必须能说出**是否通过事件中心**这个核心区别。
4. **策略**：不要只写策略对象，忘了展示**如何消除 if-else** —— 代码里要对比重构前后的差异。
